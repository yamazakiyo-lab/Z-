# 共有フォルダ整理プログラム運用マニュアル

## Ver.260607

## 1. 目的
本マニュアルは、共有フォルダ整理プログラムの日次運用、処理対象範囲、命名ルール、画像・動画処理、B4フォルダ運用、確認手順および障害時の対応方法を定めるものです。

対象環境は Windows です。

現行運用では、次の2本のデイリータスクを中心に処理を行います。

- GDX_DailyRun
- OTHER_DailyRun

現在の運用方針は、次のとおりです。

- Yドライブへの手動同期は不要とする。
- スクリプト更新は git 管理を中心とする。
- 処理対象は Zドライブ全体ではなく、実質的に 2to9_業務別フォルダ 配下を対象とする。

---

## 2. デイリータスク

### 2-1. デイリータスクの種類
日次処理は、Windows のタスクとして次の2本で実行します。

- GDX_DailyRun
- OTHER_DailyRun

### 2-2. GDX_DailyRun
GDX_DailyRun は、GDX側の整理処理を実行するタスクです。

実行経路は次のとおりです。

run_gdx_wrapper.bat  
-> run_gdx_logged.ps1  
-> run_gdx.py  
-> 91GDX・252WORKNO-program/cli.py

主に、工番別実績写真・動画、整備資料、PO LIST、工事工番に関する整理処理を行います。

### 2-3. OTHER_DailyRun
OTHER_DailyRun は、OTHER側の整理処理を実行するタスクです。

実行経路は次のとおりです。

run_91other_wrapper.bat  
-> run_91other_logged.ps1  
-> run_91other.py  
-> 91OTHER-program/cleanup.py  
-> 91OTHER-program/91フォルダ以外整理

OTHER側では、91フォルダ以外整理 の処理を通じて、対象範囲内の整理を行います。

---

## 3. 対象範囲

### 3-1. 共通方針
本プログラムは、Zドライブ全体を既定で走査するものではありません。

現行運用上の実質的な処理対象は、2to9_業務別フォルダ 配下です。

そのため、本マニュアルでは本プログラムを「Z全体を整理するプログラム」として扱いません。
2to9_業務別フォルダ 配下を中心に処理するプログラムとして扱います。

### 3-2. GDX側の対象範囲
GDX側では、主に次のフォルダを対象とします。

- 91_工番別実績写真・動画
- 252_整備資料
- 92_PO LIST
- 9781_工事工番

GDX側では、工番別の写真・動画、整備資料、PO LIST、工事工番に関するデータを対象に整理処理を行います。

### 3-3. OTHER側の対象範囲
OTHER側では、base_dir は次の設定です。

Z:\takachiho

ただし、allowed_top_dirs が次の設定であるため、実際の処理対象は 2to9_業務別フォルダ 配下に限定されます。

allowed_top_dirs = ["2to9_業務別フォルダ"]

### 3-4. allowed_top_dirs の注意点
allowed_top_dirs は、OTHER側の走査範囲を制限する重要な設定です。

allowed_top_dirs を空にすると、base_dir 全体を走査する動作になり得ます。

そのため、現行運用では allowed_top_dirs を空にしないでください。

特に、Zドライブ全体を処理対象にする意図がない場合は、次の設定を維持します。

allowed_top_dirs = ["2to9_業務別フォルダ"]

## 4. 命名ルール

### 4-1. Windows禁止文字
Windows でファイル名またはフォルダ名に使用できない文字は、処理時にアンダースコアへ置換します。

対象となる文字は次のとおりです。

\\ / : * ? " < > |

置換例は次のとおりです。

- 写真/動画 -> 写真_動画
- 出荷:確認 -> 出荷_確認
- 工番?確認 -> 工番_確認

### 4-2. 全角半角の扱い
全角半角は、一律変換する運用ではありません。

NFKC正規化は、主にフェーズ判定など、比較・分類しやすくするために使用します。

そのため、すべてのファイル名・フォルダ名を自動的に半角へ統一するものではありません。

### 4-3. アンダースコアの扱い
英単語間のアンダースコアは、補正される場合があります。

例:

before_after -> before after

ただし、連番形式や日付を含む形式は維持します。

例:

001_240101

このような連番形式は、英単語間の区切りとは扱わないため、アンダースコアを維持します。

## 5. 画像・動画処理

### 5-1. 連番リネーム
写真・動画ファイルは、日次整理処理の中で連番リネームの対象になります。

主な形式は次のとおりです。

- 001_YYMMDD
- 工番プレフィックス付きの連番形式

工番に紐づくデータでは、必要に応じて工番を先頭に付けた形式で整理します。

### 5-2. 画像圧縮
画像ファイルは、処理時に圧縮される場合があります。

圧縮上限値は、GDX側とOTHER側で異なります。

GDX側の上限値は次のとおりです。

max_kb = 1000

OTHER側の上限値は次のとおりです。

max_kb = 2048

### 5-3. 画像・動画処理時の注意
連番リネームは、写真・動画ファイルを対象に行います。

画像圧縮は、画像ファイルを対象に行います。

動画ファイルは整理対象ですが、本マニュアルでは画像圧縮の対象として扱いません。

---

## 6. B4運用

### 6-1. B4の役割
B4は、整理前写真・動画、分類前データ、またはB1/B2/B3に直接置くべきでないデータを一時的に退避・集約するためのフォルダです。

B1、B2、B3に分類できないデータや、非メディアファイルは、必要に応じてB4へ退避します。

### 6-2. B4空フォルダ削除
B4フォルダは、処理後に空になった場合、削除対象になります。

B4配下のサブディレクトリについても、中身が空になった場合は空判定を行い、削除対象になります。

### 6-3. B4配下サブディレクトリの空判定・削除
B4配下にサブディレクトリがある場合、そのサブディレクトリ内のデータを確認し、必要に応じてB1/B2/B3へ移動します。

移動後に空になったサブディレクトリは削除します。

### 6-4. junkファイルの除去
次のような不要ファイルは、junkファイルとして除去します。

- Thumbs.db
- desktop.ini
- .DS_Store

これらのファイルだけが残っている場合は、削除後に空フォルダとして扱います。

### 6-5. B1/B2/B3内の非メディア退避
B1、B2、B3内に非メディアファイルがある場合は、B4へ退避します。

これにより、B1/B2/B3は写真・動画を中心とした分類フォルダとして維持します。

## 7. 運用ポリシー

### 7-1. Yドライブ同期
現行運用では、Yドライブへの手動同期は不要です。

日次運用で、作業者が手動でYドライブへ同期する手順は必須ではありません。

そのため、通常運用、日次確認、障害確認のいずれにおいても、Yドライブへの手動同期を前提としません。

### 7-2. git管理
スクリプト更新は git 管理を中心に行います。

スクリプトを変更する場合は、変更内容を git で管理し、差分を確認できる状態にします。

運用中に個別ファイルを直接修正した場合も、最終的には git 上で変更内容を確認できるようにします。

### 7-3. 運用変更時の注意
処理対象範囲、リネーム規則、圧縮上限、B4運用、タスク実行経路を変更する場合は、マニュアルも合わせて更新します。

特に、allowed_top_dirs の変更は走査範囲に影響するため、変更時は必ず内容を確認します。

## 8. 日次確認

### 8-1. タスク状態確認
日次確認では、まず Windows のタスク状態を確認します。

確認対象は次の2本です。

- GDX_DailyRun
- OTHER_DailyRun

確認する内容は次のとおりです。

- タスクが実行されたか。
- 最終実行結果に異常がないか。
- 想定時刻に実行されているか。
- タスクが無効化されていないか。

### 8-2. 実行ログ確認
次に、実行ログを確認します。

確認対象は次のログです。

- gdx_run
- other_run

確認する内容は次のとおりです。

- ログが出力されているか。
- 実行開始と実行終了が記録されているか。
- エラーが記録されていないか。
- 対象フォルダが想定範囲内になっているか。

### 8-3. 注意ログ確認
注意ログが出ている場合は、内容を確認します。

注意ログでは、次のような内容を優先して確認します。

- 処理対象外になったフォルダまたはファイルがないか。
- 権限不足が発生していないか。
- ファイルロックにより処理できなかったものがないか。
- ネットワークドライブにアクセスできなかった記録がないか。
- 想定外のフォルダ走査が発生していないか。

### 8-4. 日次確認の基本順序
日次確認は、次の順序で行います。

1. GDX_DailyRun のタスク状態を確認する。
2. OTHER_DailyRun のタスク状態を確認する。
3. gdx_run の実行ログを確認する。
4. other_run の実行ログを確認する。
5. 注意ログを確認する。
6. エラーがある場合は、権限、ロック、ネットワークドライブ状態を優先して確認する。

## 9. 障害対処

### 9-1. タスクが実行されていない場合
GDX_DailyRun または OTHER_DailyRun が実行されていない場合は、次を確認します。

- Windows タスクが有効になっているか。
- 最終実行結果にエラーが出ていないか。
- 実行ユーザーに必要な権限があるか。
- ネットワークドライブにアクセスできる状態か。
- 実行経路上の bat、ps1、py ファイルが存在するか。

### 9-2. ログが出力されていない場合
ログが出力されていない場合は、次を確認します。

- タスク自体が起動しているか。
- ログ出力先に書き込み権限があるか。
- 実行スクリプトの途中で停止していないか。
- PowerShell スクリプトが正常に呼び出されているか。
- Python スクリプトが正常に呼び出されているか。

### 9-3. ファイルが整理されない場合
ファイルが整理されていない場合は、次を確認します。

- 対象フォルダが 2to9_業務別フォルダ 配下にあるか。
- GDX側の対象フォルダに該当するか。
- OTHER側の allowed_top_dirs により対象外になっていないか。
- ファイル名またはフォルダ名に処理対象外となる要因がないか。
- ファイルが他の利用者またはアプリケーションによりロックされていないか。

### 9-4. 画像圧縮が行われない場合
画像圧縮が想定どおりに行われない場合は、次を確認します。

- 対象が画像ファイルであるか。
- GDX側とOTHER側で上限値を取り違えていないか。
- GDX側では max_kb = 1000 であるか。
- OTHER側では max_kb = 2048 であるか。
- ファイルが破損していないか。
- ファイルがロックされていないか。

### 9-5. B4が整理されない場合
B4が整理されない場合は、次を確認します。

- B4配下のサブディレクトリにファイルが残っていないか。
- Thumbs.db などのjunkファイルだけが残っていないか。
- B1/B2/B3へ移動可能なメディアが残っていないか。
- B1/B2/B3内の非メディアファイルがB4へ退避されているか。
- フォルダまたはファイルがロックされていないか。

### 9-6. エラー時の優先確認項目
エラーが発生した場合は、次の順で確認します。

1. 権限
実行ユーザーに、対象フォルダとログ出力先への読み書き権限があるか確認します。

2. ロック
ファイルまたはフォルダが、他の利用者やアプリケーションにより使用中になっていないか確認します。

3. ネットワークドライブ状態
Z:\takachiho にアクセスできるか確認します。

4. 実行経路
bat、PowerShell、Python の順に、実行経路が正しくつながっているか確認します。

5. 対象範囲
allowed_top_dirs の設定により、意図した範囲だけを対象としているか確認します。
